# 0012. Per-change adr.md manifest over global INDEX.md

- Status: accepted, supersedes ADR-0003 and ADR-0011
- Date: 2026-06-27
- Supersedes: 0003-immutability-with-index-as-active-manifest.md, 0011-index-append-only-for-extraction.md

## Context

ADR 0003 introduced `adr/INDEX.md` as a "HEAD pointer" listing currently-in-force ADRs, avoiding the need to walk supersession chains. ADR 0011 built on this by specifying that extraction skills append-only to INDEX.md.

However, the upstream `spec-driven-with-adr` schema already has a correct mechanism: a per-change `adr.md` manifest file inside `openspec/changes/<change>/`. This manifest records which in-force ADRs were reviewed and which new ones were created — it is also the completion marker the CLI uses to know the ADR step is done.

When the schema's `generates` field was changed to point at `adr/*.md` (to find INDEX.md), the CLI matched pre-existing ADR files and reported the artifact as permanently `done`, breaking ADR review for all subsequent changes. The INDEX.md mechanism conflated per-change tracking with cross-change state.

The correct approach: in-force ADRs are derivable by reading all files in `adr/` and walking Supersedes links (exactly what the design artifact instruction already specifies). No global mutable index is needed.

## Decision

Retire `adr/INDEX.md` as the active ADR manifest. The per-change `adr.md` file (generated by the `spec-driven-with-adr` schema inside each change directory) is the sole completion marker for ADR review. Currently-in-force ADRs are determined by reading `adr/` files and computing the supersession graph — no separate index file is maintained. Extraction skills no longer append to INDEX.md.

## Consequences

- Positive: schema `generates: adr.md` works correctly as the per-change completion marker; no global mutable file to maintain
- Positive: eliminates the bookkeeping cost of keeping INDEX.md synchronized with supersession events
- Positive: extraction skills are simpler — write the ADR file, done
- Negative: determining in-force ADRs requires reading all files and walking links (slightly more expensive than a curated list, but this is what the schema instructions already require)
- Negative: existing INDEX.md files in repos are stale and confusing — they MUST be deleted during cleanup to avoid misleading Claude into thinking the mechanism still operates
- Neutral: ADR 0006's principle (process/data separation) remains valid; the "data" is now the `adr/` directory contents themselves rather than a curated INDEX.md file
